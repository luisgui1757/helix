## Design Context

### Users

Pi users who want to configure and inspect Helix without leaving the terminal or
learning an agent skill. Their primary workflow is keyboard-first: discover a
slash command, make a choice, see the resulting state, and keep working.

### Brand Personality

Native, deliberate, quiet. Helix should feel as integrated and self-explanatory
as a first-party Pi command, with the clarity and finish users expect from Codex
and Claude Code.

### Aesthetic Direction

Use Pi's active theme and TUI primitives. Favor compact lists, strong labels,
plain-language status, and immediate feedback. Do not force a Helix theme,
imitate a web dashboard, or add decorative terminal chrome.

### Design Principles

- Give each user-facing capability a discoverable `helix-*` slash command.
- Keep the common path interactive; retain concise arguments for automation.
- Show boolean state as `[x] Feature` / `[ ] Feature` and make reversible
  toggles immediate.
- Follow Pi keyboard conventions and preserve usability in narrow terminals.
- Refuse unsupported work with a stable reason and a next action.
