Helix: simple decisions and build order

A plain-language summary of what has been decided for Helix, what is blocked, and when each piece is planned.

Source: ROADMAP.md Roadmap date: 2026-07-11 Summary date: 2026-07-11 Pi pinned: 0.80.3 M0a: wrapped

One-screen answer

Current release posture: helix is the private release candidate, built from a sanitized single-root history. the legacy repository network remains a private archive. Current runtime rails are max_iterations and max_concurrency; backend billing owns spend, token counts are telemetry, and the historical fence extension is untouched/deferred.

Current v1 contract: no cost/no-spend policy, token-budget rail, write allowlist, live flag, or task-loop live transport ships. Every non-mock cast refuses as live-adapter-not-wired. The executable checks currently lock 501 top-level Node test declarations.

What Helix is

Helix is a thin, security-first layer for the Pi CLI. It should use Pi native features and audited packages before building custom extensions.

What is already decided

All core requirements stay in scope. "Minimal" means a small slash-command surface, not fewer features.

How stages are run

One larger prompt per stage (adopted 2026-07-03). After M0a, each stage is worked as a single self-contained mission — with a read-first list, an authority grant, work items, and an explicit acceptance-gate checklist — instead of many small prompts. A stage may run up to about 12 internal fix/review cycles and stops when the gates pass or a real security/architecture blocker appears.

Phases 1 and 2 may be bundled to reach the Phase 3 orchestration goal sooner (the fence, worktree manager, and review loop are its prerequisites). This is a build-order acceleration only: no planned feature is dropped, the quality bar and audits are unchanged, and deferred items (like Neovim/Pi integration) stay deferred. The stage-execution cap is separate from the runtime adversarial spend caps.
